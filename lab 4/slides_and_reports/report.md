# Теоретические сведения

## Рабочий процесс Gitflow

Gitflow Workflow представляет собой модель организации работы с системой контроля версий Git.  
Данная модель была предложена и популяризирована Винсентом Дриссеном.

Gitflow предполагает строгую структуру ветвления, ориентированную на процесс выпуска программного продукта.

Основные ветви:

- **master** — содержит стабильную версию проекта
- **develop** — основная ветка разработки

Дополнительные ветви:

- **feature** — ветви разработки новых функций
- **release** — ветви подготовки релиза
- **hotfix** — ветви исправления ошибок в релизной версии

---

## Процесс работы с Gitflow

Работа в Gitflow строится следующим образом.

Из основной ветки **master** создаётся ветка **develop**.

Из ветки **develop** создаются ветви:

- feature
- release

После завершения разработки функция объединяется обратно в ветку **develop**.

При подготовке релиза создаётся ветка **release**, которая после завершения объединяется 
с ветками **master** и **develop**.

Если возникает ошибка в рабочей версии программы, создаётся ветка **hotfix**, 
которая создаётся из ветки **master** и после исправления объединяется с ветками **master** и **develop**.

## Семантическое версионирование

Семантическое версионирование используется для управления версиями программного обеспечения.

Версия записывается в виде:

```
MAJOR.MINOR.PATCH
```

где:

- **MAJOR** — изменения, нарушающие обратную совместимость
- **MINOR** — добавление новой функциональности
- **PATCH** — исправление ошибок

Пример версии:

```
1.2.3
```

---

## Общепринятые коммиты (Conventional Commits)

Conventional Commits — это соглашение о формате сообщений коммитов.

Структура сообщения коммита:

```
<type>(<scope>): <subject>
```

Примеры типов коммитов:

- **feat** — добавление новой функциональности
- **fix** — исправление ошибок
- **docs** — изменения документации
- **style** — изменения форматирования
- **refactor** — рефакторинг кода
- **test** — изменения тестов
- **chore** — технические изменения

Использование данного соглашения позволяет автоматически формировать журнал изменений проекта.

# Практический сценарий использования Git

## Создание репозитория

Создание репозитория выполняется стандартными командами Git.

```
git init
git add .
git commit -m "first commit"
```

Подключение удалённого репозитория:

```
git remote add origin git@github.com:<username>/git-extended.git
git push -u origin main
```

## Настройка Conventional Commits

Создаётся файл **package.json**.

```
pnpm init
```

Добавляется конфигурация commitizen.

```
{
 "config": {
   "commitizen": {
     "path": "cz-conventional-changelog"
   }
 }
}
```

После этого коммиты выполняются через:

```
git cz
```

## Создание релиза

Создаём ветку релиза:

```
git flow release start 1.0.0
```

Создаём журнал изменений:

```
standard-changelog --first-release
```

Добавляем файл журнала изменений:

```
git add CHANGELOG.md
git commit -m "chore(site): add changelog"
```

Завершаем релиз:

```
git flow release finish 1.0.0
```

Отправляем изменения на сервер:

```
git push --all
git push --tags
```
