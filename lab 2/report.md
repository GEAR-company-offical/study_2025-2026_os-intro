# Теоретические сведения

## Системы контроля версий. Общие понятия

Системы контроля версий (Version Control System, VCS) применяются при работе нескольких человек над одним проектом. Обычно основное дерево проекта хранится в локальном или удалённом репозитории, к которому имеют доступ участники проекта.

При внесении изменений система контроля версий позволяет фиксировать изменения, объединять изменения разных пользователей и при необходимости возвращаться к предыдущим версиям проекта.

В централизованных системах контроля версий используется единый сервер, который хранит все версии проекта. Пользователь получает копию проекта, вносит изменения и отправляет их обратно на сервер.

Системы контроля версий также позволяют отслеживать:

- кто внёс изменения  
- когда были внесены изменения  
- какие именно изменения были сделаны  

В распределённых системах контроля версий центральный сервер не является обязательным, так как каждый пользователь хранит полную копию репозитория.

---

## Примеры использования git

Git представляет собой набор программ командной строки. Доступ к ним осуществляется через терминал с помощью команды `git`.

Git является распределённой системой контроля версий, поэтому каждая локальная копия проекта содержит полную историю изменений.

---

## Основные команды git

Создание репозитория:

```
git init
```

Получение изменений из удалённого репозитория:

```
git pull
```

Отправка изменений в удалённый репозиторий:

```
git push
```

Просмотр состояния репозитория:

```
git status
```

Просмотр изменений:

```
git diff
```

Добавление файлов:

```
git add .
```

Создание коммита:

```
git commit -m "Описание изменений"
```

Создание новой ветки:

```
git checkout -b branch_name
```

Переключение на ветку:

```
git checkout branch_name
```

Удаление ветки:

```
git branch -d branch_name
```

---

## Стандартные процедуры работы при наличии центрального репозитория

Работа начинается с получения последних изменений:

```
git checkout master
git pull
```

Создание новой ветки:

```
git checkout -b branch_name
```

После внесения изменений выполняются команды:

```
git status
git add .
git commit -m "commit message"
git push origin branch_name
```

---

## Работа с локальным репозиторием

Сначала выполняется настройка пользователя:

```
git config --global user.name "Имя Фамилия"
git config --global user.email "email@example.com"
```

Создание репозитория:

```
mkdir tutorial
cd tutorial
git init
```

Создание тестового файла:

```
echo "hello world" > hello.txt
```

Добавление файла:

```
git add hello.txt
```

Создание коммита:

```
git commit -m "Новый файл"
```

---

## Работа с сервером репозиториев

Для работы с удалённым сервером создаётся SSH ключ:

```
ssh-keygen -C "Имя Фамилия <email>"
```

После генерации ключ добавляется в настройки GitHub.

Добавление удалённого репозитория:

```
git remote add origin ssh://git@github.com/username/repository.git
```

Отправка изменений:

```
git push -u origin master
```

# Базовая настройка git

## Первичная настройка параметров git

Настройка имени и электронной почты:

```
git config --global user.name "Name Surname"
git config --global user.email "work@mail"
```

Настройка кодировки:

```
git config --global core.quotepath false
```

Настройка начальной ветки:

```
git config --global init.defaultBranch master
```

---

## Учёт переносов строк

В разных операционных системах используются разные символы перевода строк.

Windows:

```
CRLF (\r\n)
```

Linux и Unix:

```
LF (\n)
```

Настройка преобразования:

```
git config --global core.autocrlf true
```

Проверка преобразования:

```
git config --global core.safecrlf warn
```

# Создание ключа SSH

## Общая информация

SSH используется для безопасной аутентификации пользователя при работе с удалёнными репозиториями.

SSH ключи состоят из двух частей:

- приватный ключ
- публичный ключ

Публичный ключ добавляется на сервер.

---

## Создание ключа SSH

Создание ключа RSA:

```
ssh-keygen -t rsa -b 4096
```

Создание ключа Ed25519:

```
ssh-keygen -t ed25519
```

---

## Добавление SSH ключа в GitHub

Копирование ключа:

```
cat ~/.ssh/id_ed25519.pub
```

После этого ключ добавляется в настройках GitHub в разделе **SSH keys**.

---

# Верификация коммитов с помощью PGP

## Общая информация

Подпись коммитов позволяет подтвердить личность пользователя, который сделал коммит.

Для подписи используется технология PGP.

---

## Создание ключа

Генерация ключа:

```
gpg --full-generate-key
```

Параметры ключа:

- тип RSA and RSA  
- размер ключа 4096  
- срок действия не ограничен  

---

## Экспорт ключа

Просмотр ключей:

```
gpg --list-secret-keys --keyid-format LONG
```

Экспорт ключа:

```
gpg --armor --export <PGP_FINGERPRINT>
```

---

## Добавление PGP ключа в GitHub

Экспортированный ключ добавляется в настройки GitHub:

```
https://github.com/settings/keys
```

Раздел **New GPG key**.

---

## Подписывание коммитов git

Подписанный коммит:

```
git commit -S -m "commit message"
```

---

## Настройка автоматической подписи коммитов

```
git config --global user.signingkey <PGP_FINGERPRINT>
git config --global commit.gpgsign true
git config --global gpg.program gpg
```

---

## Проверка коммитов

После подписи GitHub отображает значок **Verified** рядом с коммитом.

---

## Режим бдительности (vigilant mode)

В настройках GitHub можно включить режим vigilant mode.

В этом режиме все неподписанные коммиты помечаются как **Unverified**.

# Контрольные вопросы

## 1. Что такое системы контроля версий

Система контроля версий — это программное средство, позволяющее отслеживать изменения файлов проекта и хранить историю этих изменений.

---

## 2. Понятия VCS

Хранилище — репозиторий, где хранится проект.  
Commit — сохранённое изменение проекта.  
История — последовательность коммитов.  
Рабочая копия — локальная версия проекта.

---

## 3. Централизованные и распределённые VCS

Централизованные используют один сервер. Пример — Subversion.

Распределённые позволяют каждому пользователю иметь полную копию репозитория. Пример — Git.

---

## 4. Работа при единоличной разработке

Пользователь создаёт локальный репозиторий, вносит изменения и фиксирует их с помощью коммитов.

---

## 5. Работа с общим хранилищем

Пользователь получает изменения, вносит свои изменения, делает коммит и отправляет изменения на сервер.

---

## 6. Основные задачи Git

- управление версиями файлов  
- хранение истории изменений  
- совместная работа над проектом  

---

## 7. Основные команды Git

git init  
git status  
git add  
git commit  
git push  
git pull  

---

## 8. Работа с локальным и удалённым репозиториями

Локальный репозиторий хранится на компьютере пользователя. Удалённый репозиторий находится на сервере и используется для обмена изменениями.

---

## 9. Что такое ветви

Ветви позволяют разрабатывать новые функции независимо от основной версии проекта.

---

## 10. Игнорирование файлов

Для игнорирования файлов используется файл `.gitignore`.

Пример:

```

build/
node_modules/
```
